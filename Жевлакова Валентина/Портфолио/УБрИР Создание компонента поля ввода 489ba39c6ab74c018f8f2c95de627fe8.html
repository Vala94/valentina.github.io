УБрИР. Создание компонента поля ввода

Обложка

Задача

  1. Разработать поле ввода с актуальным поведением
  1. Собрать компонент с использованием последних обновлений Figma

Проблема

  1. Старые поля ввода не отвечают современным тенденциям в отображении ошибок и подсказок
  1. Разное поведение в зависимости от сценария
  1. Компонент собран устаревшим способом и постоянно ломается

Решение

  1. Провела ресёрч конкурентов по работе полей ввода

  1. Проанализировала все сценарии и составила схему с видами полей

  1. Собрала стандартные поля ввода в единый компонент с помощью instance swap. При создании компонента использовала атомарный подход. Договорилась с разработчиками использовать единый нейминг по принципу camelCase.

Посмотрим настройку компонента

  • Первая настройка позволяет выбрать тип поля. Рассмотрим далее все на примере текстового типа поля

  • Настройка bottomPart позволяет отключать нижнюю часть компонента

  • Настройка state позволяет выбрать состояние поля. Она находится в группе настроек верхней части компонента topPartText, приставка Text говорит о том, что это текстовый тип поля

  • Настройка error включает состояние ошибки

  • Настройка icon включает иконку. При включении настройки ниже появляется дополнительная группа rightPart с настройкой иконки

  • В группе bottomPart находится настройка type для выбора типа нижней части компонента

Аналогичным образом я собрала кастомные типы полей:

  1. Поле с номером карты

  1. Поле для обмена валюты

  1. Поле с паролем

  1. Поле с выбором даты

Поведение полей ввода

Я написала 8 принципов поведения полей:

  1. Используем мгновенную валидацию при заполнении заведомо неверных значений
    Прототип
  1. При потере фокуса не валидируем поле на пустоту
    Прототип
  1. Используем валидацию при смене фокуса по формату значения (например по количеству символов)
    Прототип
  1. Используем валидацию при отправке формы когда нельзя проверить поля при потере фокуса
    Прототип
  1. При отправке формы осуществлять прокрутку к первому полю с ошибкой. Если поле оказалось не заполненным, не нужно выводить подсказку, а достаточно подсветить само поле
    Прототип
  1. При отправке формы выводим состояние ошибки для поля без подстрочника. Кнопка для отправки формы всегда в стоянии нормал (не дизейблим)
    Прототип
  1. Когда начинаешь исправлять что-то в поле ввода с него уходит состояние ошибки
    Прототип
  1. Ограничить ввод неподходящих символов. Если в поле нужно вводить только цифры, и это очевидно пользователю, нужно игнорировать ввод букв (вывод нужной клавиатуры)
